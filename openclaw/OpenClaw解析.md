OpenClaw解析





记忆模块：

+ 分为三部分
  + 回溯记忆：工作时当前对话框的记忆
  + 核心记忆：有哪些skill，user偏好记忆，缺陷记忆（总结当前模型在处理某些问题时经常遇到的问题，放置到一个markdown文档中）
  + 归档记忆：不同的对话框中的记忆的总结，最后会放入到核心记忆当中
+ 还有一层是检索层，用混合检索，可以是将记忆向量化存数据库，可以是bm25关键词检索，还可以是先找到对应层次的记忆文档，再使用某些工具functioncalling搜索关键字，再获取上下多少的文段进行增强信息。

+ system prompt部分的拼接核心

  + `SKILLS_SHAPSHOT.md`：Agent的Skills列表

  + `SOUL.md`：人格语气与边界

  + `IDENTITY.md`：名称风格与表情

  + `USER.md`: 用户画像与称呼方式，性格等

  + `AGENTS.md:` 操作指令、记忆使用规则与优先级。用来修改其它的md的边界。

    + ```
      例如：
      + 当用户的指令你做不好的话，不清楚要怎么操作的话，缺失了些什么的情况下，将内容写入MEMORY.md文件中
      + 当用户提及自己叫什么，自己的性格是什么的情况下，修改USER.md内容
      等等。
      ```

  + `MEMORY.md`：跨线程的长期记忆，Agent会根据其它的指示，把一些重要来自其他对话的、非常重要的信息提取并写入到MEMORY.md中

skill调用模块：

+ 首先大模型是无法直接获取文件内容的，所以本质上是要先进行读取文件的function calling，进行函数调用读取指定元数据中对应的文件，将内容给到大模型，这样大模型就知道skill的内容，知道接下来要怎么做了。
  + 可以辅助完成的事情，自动化完成对应的软件的安装和配置。



自我进化的核心原理：

+ 当大模型使用某个skill的时候，调用某些api，或者按照某个流程会一直走错，这时，大模型就会想是否修改api对别的api进行请求，或者说修改流程的某些步骤，然后走通了，就修改skill，从而让人们觉得好用了。
+ 有篇文章是关于大模型的基因的内容，人们每次重新打开一个新的agent，大模型都不会知道以前做了些什么，明明以前大模型解决了很多的问题，有完整的跑通流程，然而每次都需要重新试错，所以发明了大模型的基因胶囊的东西，允许agent每次在解决玩某个任务时进行总结，将开发的环境，等等写成skill，共享到网络上。项目的名称是EvoMap。使用了某个用于此的协议。
+ 再联想到有关agent的职责分布边界的问题，让agent写代码时，多agent要规范自己的边界，怎样确定边界呢？根据上下文来确定，一个agent应该拥有自己要干的事情的完整的上下文，包括代码的测试。然后有agent专门用来做审计。为什么要这么设计呢？若是不同的Agent在进行代码编写时触碰相同的内容，则会相互干扰，从而导致bug的出现。且多Agent对不同任务处理时，又要重新获取上下文信息，从而导致token的消耗以及记忆的模糊等的问题。
