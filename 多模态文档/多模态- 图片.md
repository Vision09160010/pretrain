# 切片

```python

from PIL import Image

def multimodal_photo_cut(image_path, target_size=336):
    # 打开图片
    image = Image.open(image_path)

    # 获取图片的宽度和高度
    width, height = image.size

    # 缩放图片
    gloal_img = image.resize(size=(target_size,target_size))

    #切成几分
    grid_x,grid_y = 2,2
    step_x,step_y = width//2,height//2

    # 图片集
    patches = []

    # 切图 先y后x，符合视觉习惯，一行看完再看下一行
    for i in range(grid_y):
        for j in range(grid_x):
            # 拿到图片的四个点
            box = (i*step_x, j*step_y, (i+1)*step_x, (j+1)*step_y)
            # 裁剪图片
            patch = image.crop(box).resize(size=(target_size,target_size))
            # 加入图片集
            patches.append(patch)

    return gloal_img,patches

if __name__ == '__main__':
    gloal_img,patches = multimodal_photo_cut('./Multimodal_photo.png')
    print(patches)
    for patch in patches:
        patch.show()
    # gloal_img.show()
    
```


## 为什么要先Y 后 X

这是一个非常经典且触及计算机视觉底层逻辑的好问题。

之所以采用“**先 y (行/高) 后 x (列/宽)**”的循环方式（即 `for i in range(grid_y): for j in range(grid_x):`），主要基于以下三个核心原因：

### 1. 符合人类和 LLM 的“阅读顺序” (Raster Scan Order)

这是最直接的原因。

*   **人类习惯**：在这个星球上，绝大多数语言的阅读顺序是 **“从左到右，从上到下”**。
*   **LLM 的习惯**：大模型本质上是一个预测下一个 Token 的序列模型。当你把一张二维图片切成一串 Patch 序列喂给它时，它默认这段序列的空间逻辑也是遵循 **“Z字形扫描” (Raster Scan / Z-order)** 的。

**演示对比：**
假设我们将图片切成 $2 \times 2$ 的网格：
```text
[Patch 0] [Patch 1]
[Patch 2] [Patch 3]
```

*   **先 Y 后 X (行优先 - 正确)**:
    序列顺序：`Patch 0` -> `Patch 1` -> `Patch 2` -> `Patch 3`
    *这符合“第一行读完，读第二行”的逻辑。*

*   **先 X 后 Y (列优先 - 错误)**:
    序列顺序：`Patch 0` -> `Patch 2` -> `Patch 1` -> `Patch 3`
    *这就变成了“先读左边这一竖条，再读右边那一竖条”。这会打乱 LLM 对空间位置的理解（Position Embedding），导致模型“看晕了”。*

### 2. 符合计算机内存的物理布局 (Row-Major Order)

在 Python (NumPy/PyTorch) 和 C/C++ 中，多维数组默认是 **行优先存储 (Row-Major)** 的。

*   **内存视角**：
    图片在内存里并不是二维的，而是一条长长的连贯的一维数据。
    比如一张 $2 \times 2$ 的图，内存里是这样存的：`[行0列0, 行0列1, 行1列0, 行1列1]`。

*   **效率**：
    *   **先遍历 Y (行)**: 意味着你锁定了第一行，然后快速扫过这一行的所有像素（X）。这在内存上是**连续访问**的，能最大化利用 CPU/GPU 的**缓存 (Cache)**，速度最快。
    *   **先遍历 X (列)**: 意味着你取了第一行的第1个点，然后跳过几千个点去取第二行的第1个点。这种**跳跃式访问 (Strided Access)** 极易导致 Cache Miss，大幅降低处理效率。

### 3. 矩阵与图像的坐标定义惯例

在数学矩阵和图像处理代码中，坐标系的定义通常是：
*   **Shape**: `(Height, Width)` -> `(Y, X)`
*   **Indices**: `image[row_index][col_index]` -> `image[y][x]`

```python
# 假设 grid_y = 2 (两行), grid_x = 3 (三列)

for i in range(grid_y):      # i 代表第几行 (Row / Height / Y)
    for j in range(grid_x):  # j 代表第几列 (Col / Width / X)
        
        # 计算切片的坐标框 Box
        # Box = (Left, Upper, Right, Lower) -> (x1, y1, x2, y2)
        
        y_start = i * step_h
        x_start = j * step_w
        
        # PIL 的 crop 需要 (x_start, y_start, ...)
        # 但循环逻辑决定了我们是“一行一行”地切
```

## 总结

*   **为了让模型读得懂**：必须按“从左到右，从上到下”的顺序排列 Patch。
*   **为了代码跑得快**：顺应内存的“行优先”布局。

所以，在所有涉及图像处理（无论是切片、卷积还是像素遍历）的代码中，你几乎永远都会看到 **Outer Loop = Height (y), Inner Loop = Width (x)** 的写法。


# 论文

先结合预训练的T5（Raffel等，2020）和ViT（Dosovitskiy等，2020）模型构建一个骨干编码器（见图3），该编码器能够将图文对、仅图像以及仅文本输入编码为多模态表示。


这篇论文提出了一种名为 **MuRAG (Multimodal Retrieval-Augmented Generator)** 的模型。简单来说，它是经典的 RAG（检索增强生成）模型的**多模态升级版**。

传统的 RAG 模型只能从外部知识库中检索**文本**来辅助回答问题，而 MuRAG 能够从外部记忆库中检索**图像和文本**，从而回答那些需要视觉知识才能解决的问题（例如：“在这个图片的背景里有什么？”）。

以下我将从**核心原理（模型架构）**和**工作流程（训练与推理）**两个维度为您详细拆解。

---

## 一、 核心原理与架构 (Model & Principles)

MuRAG 的核心思想是将“检索”和“生成”结合，并在多模态空间中进行。它主要由三个部分组成：**骨干编码器 (Backbone Encoder)**、**检索器 (Retriever)** 和 **阅读生成器 (Reader/Generator)**。

### 1. 骨干编码器 (Backbone Encoder)
这是理解图像和文本的基础。MuRAG 并没有从头训练，而是利用了预训练好的模型：
*   **视觉部分**：使用 **ViT (Vision Transformer)**，将图像切分为 $16 \times 16$ 的补丁（Patches），编码成序列向量。
*   **文本部分**：使用 **T5 (Text Transformer)** 的编码器，将文本 Token 转换为嵌入向量。
*   **融合**：图像和文本的嵌入向量被拼接在一起，输入到一个双向 Transformer 中进行交互，最终形成多模态表示。

### 2. 检索器 (Retriever)
检索器的作用是在巨大的外部知识库（Memory）中找到与当前问题最相关的信息。
*   **双塔结构**：使用骨干编码器分别对“查询（Query）”和“记忆项（Memory candidates）”进行编码。
*   **[CLS] Token**：在编码序列前加一个 `[CLS]` token，取该位置的输出向量作为整个输入（查询或记忆项）的**全局表示**。
*   **检索机制**：使用 **MIPS (最大内积搜索)** 计算查询向量与记忆库中所有项的向量点积，找出 Top-K 个最相似的项（可能是图片，也可能是文本段落）。

### 3. 阅读生成器 (Reader)
检索到相关信息后，需要阅读这些信息并生成答案。
*   **输入增强**：将检索到的 Top-K 个项（原始图像补丁或文本）与原始查询 $q$ 拼接在一起，形成一个增强的输入序列 $[m_1, ..., m_k, q]$。
*   **生成**：将这个长序列输入到骨干编码器中进行编码，然后使用 T5 的**解码器 (Decoder)** 逐个 Token 地生成最终的文本答案。

---

## 二、 工作流程 (Process)

MuRAG 的流程分为**预训练 (Pre-training)** 和 **微调 (Fine-tuning)** 两个主要阶段。

### 1. 预训练阶段 (Pre-training)
目的是让模型学会如何检索多模态信息，并利用这些信息回答问题。

*   **数据集**：混合了图文对数据（如 LAION, CC）和纯文本问答数据（如 PAQ, VQA）。
*   **输入构建**：
    *   如果是图文数据：输入图片作为“查询”，目标是生成对应的描述文本（Caption）。
    *   如果是问答数据：输入问题，目标是生成答案。
*   **记忆库 (Memory)**：为了避免在海量全量数据上进行反向传播带来的巨大开销，MuRAG 在预训练阶段采用了 **In-Batch Memory（批次内记忆库）**。
    *   即：对于当前 Batch 中的每一个样本，其正确答案（正例）存在于 Batch 内，而 Batch 内的其他样本作为负例（干扰项）。
*   **损失函数 (Loss)**：联合优化两个损失：
    *   **$L_{con}$ (对比损失)**：训练检索器。让查询向量与正确的记忆项向量距离更近，与错误的距离更远。
    *   **$L_{gen}$ (生成损失)**：训练生成器。在给定检索内容的前提下，最大化生成正确答案的概率。

### 2. 微调阶段 (Fine-tuning)
将预训练好的模型迁移到下游任务（如 WebQA 或 MultimodalQA）。这是一个两阶段过程：

*   **第一阶段：In-Batch Training (批次内训练)**
    *   原理同预训练，主要让模型适应下游任务的数据格式（输入是文本问题，检索目标是图文对）。
    *   此时还未连接全局的大规模知识库，仅在 Batch 内部做检索训练。

*   **第二阶段：Fixed-Retrieval Training (固定检索训练)**
    *   **构建索引**：利用训练好的编码器，对所有的外部知识（百万级的图片和文本）进行编码，建立全局索引。
    *   **全局检索**：在训练时，不再局限于 Batch 内部，而是通过 MIPS 在全局索引中检索 Top-K。
    *   **固定检索器**：在这个阶段，**不再更新编码器（检索器）的参数**，只更新生成器（解码器）的参数。这样做是为了节省计算资源，因为每次更新编码器都需要重新构建百万级的索引，成本太高。

### 3. 推理/预测 (Inference)
当用户提出一个问题（例如：“白宫的阳台上有什么？”）：
1.  **编码**：模型将问题编码为向量。
2.  **检索**：在构建好的多模态知识库中搜索，找到了“白宫圣诞节”的相关图片和新闻片段。
3.  **阅读**：模型“看”到了图片中阳台上的花环，“读”到了新闻里的描述。
4.  **生成**：解码器结合这些信息，生成答案：“Wreaths and garlands（花环和花冠）”。

## 三、 总结 MuRAG 的突破点
1.  **多模态检索**：打破了以往 RAG 只能检索文本的限制，能够利用海量的非结构化图像数据作为知识源。
2.  **统一架构**：在同一个 Transformer 框架下同时处理检索和生成。
3.  **性能卓越**：在 WebQA 和 MultimodalQA 等数据集上，比之前的基线模型（如 CLIP+生成模型）准确率提升了 10-20%，达到了 SOTA（State-of-the-Art）水平。

# 微调

在多模态场景下，仅有“文本问答对（Q+A）”是不够的。要复现或参考 MuRAG 的微调流程，你需要将数据升级为**“多模态三元组/四元组”**，并采用**“检索+生成”联合优化**的策略。

基于 MuRAG 论文（特别是 **3.4 Fine-tuning** 章节），我为您总结了多模态微调的核心方法论：

## 1. 数据准备：从 (Q, A) 升级到多模态格式

在纯文本微调中，你的数据是 `Question -> Answer`。
在多模态 RAG 微调中，你需要构建 **(Query, Positive_Evidence, Hard_Negatives, Answer)**。

*   **Query ($x_q$)**: 用户的文本问题（例如：“图中这辆车的型号是什么？”）。
*   **Positive Evidence ($m^+$)**: 能够回答该问题的**图片**或**图片+文字描述**（这是模型必须检索到的正确知识）。
*   **Hard Negatives ($m^-$)**: **难负例**。这是非常关键的一步。
    *   *定义*：那些看起来和问题很像（比如图片里也有车），但实际上是错误的（型号不对）的图片。
    *   *论文做法*：MuRAG 使用了 Bing Search API 或 Wikipedia 挖掘的相似但不相关的图文作为负例。
*   **Answer ($y$)**: 最终的文本回答。

**数据格式示例：**
```json
{
  "query": "白宫圣诞节阳台上有什么？",
  "pos_evidence": ["image_whitehouse_xmas.jpg", "caption: 白宫阳台装饰着花环"],
  "neg_evidence": ["image_whitehouse_summer.jpg", "caption: 白宫夏日的草坪"],
  "answer": "花环和花冠"
}
```

---

## 2. 微调的核心原理：联合损失函数 (Joint Loss)

MuRAG 的微调之所以有效，是因为它**同时**训练了“找得准（检索）”和“答得对（生成）”。

你需要设计一个联合 Loss，公式如下：
$$L = L_{con} + L_{gen}$$

*   **$L_{con}$ (Contrastive Loss - 对比损失)**：
    *   **目的**：训练**检索器**（Backbone Encoder）。
    *   **操作**：拉近 `Query` 向量与 `Positive Evidence` 向量的距离，推远与 `Hard Negatives` 的距离。
    *   **意义**：这一步教会模型：“当用户问这个问题时，你应该去看这张图，而不是那张图。”

*   **$L_{gen}$ (Generative Loss - 生成损失)**：
    *   **目的**：训练**生成器**（Decoder）。
    *   **操作**：将检索到的 Top-K 图片特征（即使检索得不完美）和 Query 拼在一起输入 Decoder，计算生成 `Answer` 的交叉熵损失（Cross-Entropy Loss）。
    *   **意义**：这一步教会模型：“看着这张检索回来的图，组织语言回答用户的问题。”

---

## 3. 两阶段微调流程 (Two-Stage Pipeline)




基于 MuRAG 论文及多模态 RAG 的最佳实践，**“两阶段微调（Two-Stage Fine-tuning）”** 是解决多模态检索与生成协同训练的核心策略。

这个流程的设计初衷是为了解决一个**工程矛盾**：**“训练检索器需要大 Batch 和全量负例”** vs **“训练生成器需要高显存和复杂计算”**。



---

### 第一阶段：In-Batch Training (批次内微调)

*   **痛点**：每次更新参数都要重新对百万级数据库建索引，太慢了。
*   **做法**：只在当前的 Batch（比如 64 条数据）内部做检索。
    *   对于第 1 个问题，它的正例是第 1 张图。
    *   它的负例是 Batch 里的其他 63 张图（加上显式挖掘的难负例）。
*   **目的**：快速让模型适应“图文检索”和“看图答题”的模式。

**核心目标**：让模型学会“该看哪张图”和“怎么看图说话”。
**计算状态**：**动态检索**（不需要预先建立百万级索引）。

#### 1. 原理
在这个阶段，我们**不连接**那个庞大的外部知识库（百万/千万级数据）。所有的“知识”都来自当前的 **Batch（批次）** 内部。

*   假设 Batch Size = $B$（例如 64）。
*   对于第 $i$ 个问题 $Q_i$，它的**正例（Positive）**是同在 Batch 里的第 $i$ 张图/文 $M_i$。
*   它的**负例（Negatives）**是 Batch 里剩下的 $B-1$ 张图/文（即 $M_j, j \neq i$）。

#### 2. 训练过程
*   **输入**：一个 Batch 的 `(Question, Image/Text, Answer)` 数据。
*   **检索训练 ($L_{con}$)**：
    *   模型计算 $Q_i$ 和 Batch 内所有 $M$ 的相似度。
    *   目标是让 $Q_i$ 和 $M_i$ 的相似度最高。
    *   *这教会了 Encoder（检索器）：在 64 个选项里，哪一个是正确答案。*
*   **生成训练 ($L_{gen}$)**：
    *   模型拿到 $M_i$（可能还有检索错的 Top-K），拼接上 $Q_i$。
    *   Decoder 负责生成 $Answer$。
    *   *这教会了 Decoder（生成器）：看着这张图，把答案写出来。*

#### 3. 优点与局限
*   ✅ **优点**：极快，不需要反复重建索引，梯度可以同时回传给 Encoder 和 Decoder。
*   ❌ **局限**：视野太窄。模型只学会了在 64 个选项里做选择题，还没见过“大海捞针（从百万数据里找）”的场面。

---

### 第二阶段：Fixed-Retrieval Training (固定检索微调)

*   **痛点**：In-Batch 视野太小，模型没见过“大海捞针”的场面。
*   **做法**：
    1.  利用第一阶段训练好的 Encoder，对所有数据库（百万级图片）进行编码，建立**全局索引**。
    2.  **冻结检索器**（Encoder 参数不更新），只训练**生成器**（Decoder）。
    3.  对于每个 Query，去全局索引里搜 Top-K（模拟真实环境），把搜到的图喂给 Decoder 训练生成。
*   **目的**：让生成器学会处理真实的（可能包含噪音的）全局检索结果，进一步提升回答准确率。

**核心目标**：让生成器适应“真实世界”的检索噪音，模拟上线后的状态。
**计算状态**：**静态检索**（Encoder 冻结，只练 Decoder）。

#### 1. 准备工作（建索引）
在进入第二阶段前，利用**第一阶段训练好**的 Encoder，对整个外部知识库（例如 1000 万张图片）进行一次全量编码，存入向量数据库（如 Milvus/Faiss）。

#### 2. 训练过程
*   **冻结 Encoder**：这时候检索器已经不准动了（也就是 **Fixed-Retrieval** 的含义）。如果动了 Encoder，向量空间就变了，索引就得重在大规模数据上重建，成本太高。
*   **全局检索**：
    *   对于问题 $Q_i$，去那个 1000 万级的索引里真的搜一遍。
    *   搜回来的 **Top-K** 结果中，可能包含正例，也可能**全是错的（负例）**，或者包含很多干扰项。
*   **生成训练 ($L_{gen}$ Only)**：
    *   把这 Top-K 个（可能是包含噪音的）结果喂给 Decoder。
    *   强迫 Decoder 即使在有干扰的情况下，也要生成正确的 $Answer$。

#### 3. 优点
*   ✅ **真实模拟**：这是模型上线后真正面对的场景（从海量数据里搜）。
*   ✅ **抗噪能力**：生成器学会了辨别检索回来的结果是否有用，如果检索错了，它要学会“瞎编”或者依靠内部知识（Parametric Knowledge）来补救，而不是被错误的检索结果带偏。

---

### 总结：为什么不直接一步到位？

你可能会问：*“为什么不直接在全量索引上做端到端训练（E2E Training）？”*

**答案是算力成本爆炸。**
如果在全量数据上做端到端训练，意味着每更新一次参数，你都要重新刷新 1000 万张图片的向量索引（因为 Encoder 变了，向量就变了）。这在工程上几乎是不可行的（除非你有 Google/Meta 级别的几千张 TPU）。

**MuRAG 的两阶段方案是工业界公认的“最优解”：**
1.  先在小圈子（In-Batch）里把**检索器**练好。
2.  锁住检索器，再去大世界（Global Index）里把**生成器**练皮实。

## 4. 专家实战建议：针对您的场景

如果您之前的“问答对”主要是文本，现在想转多模态微调，您面临的主要挑战是**“正例图片的缺失”**。

**解决方案：**

1.  **构造合成数据 (Synthetic Data Generation)**：
    *   如果您有行业图片库（比如医疗影像、电商商品图），利用 **GPT-4o** 或 **Qwen-VL** 等强大的 VLM。
    *   **Prompt**: *"请根据这张图片生成 5 个‘问题-答案’对，问题需要涉及图片细节。"*
    *   这样您就从 `Image` 反向生成了 `(Q, Image, A)` 数据对。

2.  **挖掘难负例 (Hard Negative Mining)**：
    *   在微调前，先用基础的 CLIP 或 BGE-Visualized 模型，拿你的 Query 去图片库里搜。
    *   搜出来排在前面、但**不是**正确答案对应图片的那些图，就是最好的训练素材（Hard Negatives）。

**总结一句：**
多模态微调的本质，就是把**“问题-答案”**的映射，拉伸为**“问题-图片-答案”**的链路，并通过**对比学习（找图）**和**语言建模（答题）**两个任务来同时优化这条链路。


# 微调

利用强化学习（RL）对多模态大模型（VLM）进行微调，是目前（2024-2025年）解决 **“视觉幻觉（Visual Hallucination）”** 和提升 **“复杂推理能力”** 的最前沿技术路线。

这通常被称为 **RLHF-V (RLHF for Vision-Language Models)**。

与纯文本 RL 不同，多模态 RL 的核心难点在于：**如何定义“看着图说话”的奖励（Reward）？**

以下是作为架构专家，为您设计的**多模态强化学习微调方案**，涵盖了**算法选择**、**数据构建**和**奖励设计**三个核心维度。

---

## 一、 算法选择：DPO vs PPO

在多模态场景下，**DPO (Direct Preference Optimization)** 是目前的绝对主流，而 PPO 更多用于需要与环境交互（如 Web Agent）的场景。

### 1. 方案 A：DPO (直接偏好优化) —— **推荐首选**
*   **适用场景**：减少幻觉（Hallucination）、提升回答的安全性、对齐人类审美。
*   **原理**：不需要训练独立的 Reward Model。直接构建 `(Image, Prompt, Winner, Loser)` 数据对，通过数学推导直接优化 Policy。
*   **为什么选它**：多模态模型显存占用极大，PPO 需要同时加载 4 个模型（Actor, Critic, Ref, Reward），显存容易爆。DPO 只需要加载 2 个（Policy, Ref），且更稳定。

### 2. 方案 B：PPO / GRPO (在线强化学习)
*   **适用场景**：多步视觉推理（Visual Reasoning）、Web Agent（点击网页）、CoT（思维链）。
*   **原理**：模型生成一步，环境（或 Reward Model）给一个分。
*   **GRPO (DeepSeek-R1 方案)**：这是最新的趋势。不训练 Critic，而是让 VLM 对同一张图生成一组答案，计算组内相对优势。这对于训练 VLM 进行复杂的数学/几何推理非常有效。

---

## 二、 数据构建：怎么造“多模态偏好数据”？

这是最关键的一步。你需要构建三元组：**[图片 + 问题] -> [好回答 (Chosen)] vs [坏回答 (Rejected)]**。

### 1. 针对“幻觉消除” (Hallucination Reduction)
这是 VLM 最需要 RL 的地方。SFT 后的模型经常会“看图瞎编”。
*   **数据构造逻辑**：
    *   **Image**: 一张包含“红苹果”的图。
    *   **Prompt**: “图里有什么水果？”
    *   **Chosen (Winner)**: “这是一个红苹果。” (Factually Correct)
    *   **Rejected (Loser)**: “这是一个红苹果和一个香蕉。” (Hallucination - 出现了图里没有的物体)
*   **自动化挖掘**：使用 GPT-4o 检测 VLM 的输出是否与 Object Detection（目标检测）的结果一致，不一致的作为 Loser。

### 2. 针对“详细描述” (Detailed Captioning)
*   **数据构造逻辑**：
    *   **Chosen**: 逻辑清晰、覆盖了图片主要细节、没有车轱辘话的描述。
    *   **Rejected**: 只有简短一句话，或者重复啰嗦的描述。

---

## 三、 奖励设计 (Reward Engineering)

如果你使用 PPO/GRPO，或者需要构造 DPO 数据，你需要定义“什么是好”。

#### 1. 基于规则的奖励 (Rule-based Reward)
*   **OCR 匹配**：如果问题是“图中的文字是什么？”，答案必须包含 OCR 引擎提取出的关键词。
*   **Grounding 匹配**：如果模型输出了 Bounding Box `[x1, y1, x2, y2]`，计算该框与 Ground Truth 框的 **IoU (交并比)**。IoU > 0.5 给正奖励，否则给负奖励。

### 2. 基于模型的奖励 (Model-based Reward)
*   **LLM-as-a-Judge**：把图片（转文本描述）和生成的答案喂给 GPT-4 或更强的 VLM 打分。
    *   *Prompt*: "请对比这张图的内容和模型的回答。如果回答中包含了图中不存在的物体，给 -10 分；如果完全正确，给 +10 分。"

---

## 四、 实战流程 (Implementation Pipeline)

假设我们使用 **LLaVA / Qwen-VL** 作为基座，使用 **QLoRA + DPO** 进行微调。

### Step 1: 准备数据 (JSONL)
格式通常如下（兼容 `TRL` 库）：
```json
{
  "images": ["/path/to/img.jpg"],
  "prompt": "这张图里的人在干什么？",
  "chosen": [{"role": "assistant", "content": "图中的人在骑自行车。"}],
  "rejected": [{"role": "assistant", "content": "图中的人在开车。"}]
}
```

### Step 2: 编写训练脚本 (使用 TRL + DeepSpeed)
目前 HuggingFace 的 `TRL` (Transformer Reinforcement Learning) 库已经完美支持多模态 DPO。

```python
from trl import DPOConfig, DPOTrainer
from transformers import AutoModelForVision2Seq

# 1. 加载模型 (开启 QLoRA)
model = AutoModelForVision2Seq.from_pretrained(
    "Qwen/Qwen-VL-Chat", 
    load_in_4bit=True, 
    device_map="auto"
)
# Load Ref Model (通常是同一个模型的 Copy，或者使用 PEFT 共享基座)

# 2. 配置 DPO
dpo_config = DPOConfig(
    output_dir="./dpo_results",
    beta=0.1,              # 控制与基座模型的偏离程度
    learning_rate=5e-6,    # RL 的学习率要非常小
    per_device_train_batch_size=2,
    gradient_accumulation_steps=8,
    remove_unused_columns=False # 多模态数据必须设为 False
)

# 3. 启动训练
trainer = DPOTrainer(
    model=model,
    ref_model=None, # 使用 PEFT 时不需要显式加载 Ref Model
    args=dpo_config,
    train_dataset=dataset,
    tokenizer=tokenizer,
    peft_config=peft_config # 你的 QLoRA 配置

trainer.train()
```

## 五、 在您的 Agent / MuRAG 项目中的应用

回到您之前的 **MuRAG / Agent** 项目，强化学习可以解决以下痛点：

1.  **拒绝回答 (Refusal Learning)**：
    *   *场景*：检索器检索回来的图片跟问题完全无关（检索失败）。
    *   *SFT 表现*：SFT 模型通常会强行编造一个答案。
    *   *RL 目标*：训练模型在这种情况下输出“根据检索到的图片，无法回答该问题”。
    *   *构造数据*：`(图不对题, 问题, "我不知道", "图里有...")`。

2.  **多步 Agent 规划**：
    *   *场景*：Agent 操作网页。
    *   *RL 目标*：如果 Agent 点击了正确的按钮（进入了下一步），给 Reward；如果报错，给 Penalty。
    *   *算法*：这里必须用 **PPO**，因为这是多步决策过程。

## 总结
要在多模态中做微调：
1.  **入门/去幻觉**：直接上 **DPO**。构造“正误图文对”数据是核心，代码用 `TRL` 库几行就能跑起来。
2.  **进阶/Agent**：使用 **PPO**。将环境反馈（是否报错、任务是否完成）作为 Reward。
3.  **核心心法**：多模态 RL 的本质是**教会模型“诚实”**——看见了就是看见了，没看见别瞎编。
