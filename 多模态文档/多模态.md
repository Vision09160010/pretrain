# 怎么输入

终极目标：将所有形式都变成向量

也就是所谓的模型对齐，将所有的媒介都转为和文本Embedding一样的向量，进入transform

## 图片输入

- **处理流程**:
    
    1. **切片 (Patching)**: 将图片（如 336×336）切成 14×14的小块（Patches）。
        
    2. **视觉编码 (Visual Encoder)**: 使用预训练好的 Vision Transformer (如 CLIP-ViT-L/14, SigLIP, DINOv2)。
        
    3. **特征提取**: 得到形状为 [Batch, Num_Patches, ViT_Dim] 的特征图。
        
    4. **投影 (Projection)**: 使用 MLP 或线性层 将 ViT_Dim 转换为 LLM_Dim。
        
    5. **特殊 Token**: 通常在图片特征前后加上 <image_start> 和 <image_end> 标记。

​	**输入形式 (Prompt 层面)**:
```
User: <image_start>[Image Embeddings]<image_end>\n这张图里有什么？
```


### 音频输入 (Audio Input)

音频处理的难点在于**长度可变性**和**信息密度**（如 Qwen-Audio, Salmonn）。

- **处理流程**:
    
    1. **预处理**: 将 PCM 波形转换为**梅尔频谱图 (Mel-spectrogram)**。
        
    2. **音频编码器 (Audio Encoder)**: 使用 Whisper 编码器、HuBERT 或 BEATs。
        
    3. **下采样/压缩**: 音频序列通常很长，需要通过卷积（Conv1d）或 Q-Former 进行下采样，减少 Token 数量。
        
    4. **投影**: 映射到 LLM 空间。
        
- **输入形式**:
    
    ```
    User: <audio_start>[Audio Embeddings]<audio_end>\n这段录音的情感是怎样的？
    ```
    
- **技术细节**:
    
    - **Whisper Encoder**: 擅长语音识别（ASR）任务。
        
    - **CLAP**: 擅长非语音的声音识别（如狗叫、爆炸声）。
        
    - **Windowing**: 对超长音频进行滑动窗口切片处理。
        

### 4. 视频输入 (Video Input)

视频可以被视为**带有时间维度的图片序列**（如 Video-LLaVA, LLaMA-VID）。

- **处理流程**:
    
    1. **抽帧 (Frame Sampling)**: 无法处理每一帧。通常均匀抽取 8-64 帧。
        
    2. **单帧编码**: 对每一帧复用**图片编码器**（Image Encoder）。
        
    3. **时序聚合 (Temporal Aggregation)**：
        
        - **方案 A (Mean Pooling)**: 简单地对多帧特征求平均（丢失部分时序动作信息）。
            
        - **方案 B (Time Embeddings)**: 为每一帧加上时间位置编码。
            
        - **方案 C (3D Conv/Attention)**: 使用时空注意力机制提取动作特征。
            
    4. **展平**: 将 [Frames, Patches, Dim] 展平成长序列。
        
- **输入形式**:

    
    ```
    User: <video_start>[Frame1 Emb][Frame2 Emb]...[FrameN Emb]<video_end>\n这个视频里的人在做什么动作？
    ```
    
- **关键挑战**: 显存爆炸（OOM）和上下文窗口限制。


### 原生多模态 (Native Multimodality)

上述方法叫 **Composite (组合式)** 架构（如 LLaVA），即“视觉塔 + 语言塔”。  
最新的趋势（如 GPT-4o, Chameleon, Emu）是 **Native (原生)** 架构：

- **原理**: 模型不使用独立的 Vision Encoder，而是直接训练 Transformer 接受**离散化的图像 Token**。
    
- **Tokenization**: 使用 VQ-VAE 或 VQ-GAN 将图片切块并量化成整数 Codebook ID。
    
- **输入**: 图片和文本一样，直接变成了整数 ID 序列输入模型。
    
- **优势**: 模型可以真正理解图像生成的逻辑，不仅能看，还能生成图像（Text-to-Image），且模态间的信息损耗更低。
    

### 7. 总结与建议

如果你想自己开发或微调多模态模型，建议从以下路径入手：

|   |   |   |   |
|---|---|---|---|
|模态|推荐基座 (Encoder)|推荐架构|关键技术点|
|**图片**|SigLIP / CLIP-ViT-L|LLaVA-NeXT / Qwen-VL|动态分辨率切图 (AnyRes)，MLP Projection|
|**音频**|Whisper-v3|Qwen-Audio / Salmonn|窗口化处理长音频，对齐 ASR 文本|
|**视频**|SigLIP (逐帧)|Video-LLaVA|抽帧策略，时序 Pooling，显存优化|

**工程提示**:

- **显存优化**: 多模态输入会导致序列长度激增（一张 336 的图可能占 576 tokens）。在训练时务必使用 **Flash Attention 2** 和 **Gradient Checkpointing**。


# 怎么多模态储存

多模态数据的储存并不是简单地把文件扔进硬盘。在 LLM 和 AI 应用的场景下，**多模态储存的核心挑战在于：既要存得下（海量非结构化数据），又要找得到（基于语义的快速检索）。**

通常，我们需要采用 **“三层储存架构”** 或者是 **“多模态数据库（Multimodal DB）”** 的解决方案。

以下是工业界最主流的 **多模态储存架构设计方案**：

---

### 1. 核心架构：三层储存策略 (The 3-Layer Strategy)

在生产环境中，我们绝不会把图片或视频的 Base64 编码直接存入数据库。标准的做法是将**原始文件**、**元数据**和**语义向量**分离存储。

| 层级 | 存储内容 | 推荐技术栈 (Tech Stack) | 作用 |
| :--- | :--- | :--- | :--- |
| **L1: 对象存储层** | 原始文件 (Raw Data)<br>JPG, WAV, MP4 | **MinIO (自建)**, AWS S3, Aliyun OSS | 成本最低，容量无限。负责“物理存储”。只对外提供 URL 或 Path。 |
| **L2: 结构化数据层** | 元数据 (Metadata)<br>文件名, 拍摄时间, 作者, URL | **PostgreSQL**, MongoDB, MySQL | 负责精确过滤。例如：“查找 2024年 的所有图片”。 |
| **L3: 向量存储层** | 语义向量 (Embeddings)<br>`[0.12, -0.98, ...]` | **Milvus, Qdrant, Weaviate**, pgvector | 负责语义检索。例如：“查找一只在草地上奔跑的狗”。 |

---

### 2. 深度解析：各模态的具体存法

#### (1) 图片 (Images)
*   **存储**: 图片文件存入 OSS（对象存储）。
*   **索引**: 使用 CLIP 或 SigLIP 模型将图片编码为向量（如 512维），存入向量库。
*   **关联**: 向量库中存储 `image_id`，通过 ID 在 SQL 库中找到 OSS 的 URL。

#### (2) 视频 (Videos) —— 最复杂
视频不能只存一个向量，因为视频包含时间维度。
*   **方案 A (粗粒度)**: 对视频做 Mean Pooling，存一个总向量（适合短视频分类）。
*   **方案 B (细粒度 - 推荐)**: **切片存储 (Chunking)**。
    *   每隔 5 秒或检测场景变换（Scene Detection）截取关键帧。
    *   对每个关键帧生成向量。
    *   在向量库中存储：`{vector, video_id, timestamp_start, timestamp_end}`。
    *   **检索时**: 可以定位到“视频第 3 分 20 秒出现了爆炸”。

#### (3) 音频 (Audio)
*   **存储**: 原始 WAV/MP3 存入 OSS。
*   **处理**:
    1.  **ASR 转录**: 将音频转为文字（Transcript），存入 Elasticsearch 或 SQL 用于全文检索。
    2.  **声纹/语义向量**: 使用 CLAP 或 Wav2Vec 生成向量，用于检索声音特征（如“找一段下雨的声音”）。

---

### 3. 代码实战：构建一个多模态储存流

这里是一个基于 Python 的完整伪代码流程，展示如何组合 **MinIO (对象存储)** + **Qdrant (向量存储)** + **Postgres (元数据)**。

```python
import os
import uuid
from datetime import datetime
from qdrant_client import QdrantClient
from qdrant_client.models import PointStruct
import psycopg2
import boto3 # 用于连接 MinIO/S3

class MultimodalStorage:
    def __init__(self):
        # 1. 初始化对象存储 (MinIO)
        self.s3 = boto3.client('s3', endpoint_url='http://minio:9000', ...)
        self.bucket = "multimodal-raw"
        
        # 2. 初始化向量数据库 (Qdrant)
        self.qdrant = QdrantClient("localhost", port=6333)
        self.collection = "visual_search"
        
        # 3. 初始化元数据数据库 (Postgres)
        self.pg_conn = psycopg2.connect("dbname=meta user=postgres ...")
        
        # 4. 加载编码模型 (假设已有 CLIP)
        # self.encoder = CLIPModel.from_pretrained(...)

    def save_image(self, image_path, metadata):
        """
        核心流程：上传文件 -> 提取向量 -> 存入 SQL -> 存入 VectorDB
        """
        # 生成唯一 ID
        asset_id = str(uuid.uuid4())
        file_ext = image_path.split('.')[-1]
        s3_key = f"{datetime.now().year}/{asset_id}.{file_ext}"
        
        # --- Step 1: 上传原始文件到对象存储 ---
        with open(image_path, "rb") as f:
            self.s3.upload_fileobj(f, self.bucket, s3_key)
        s3_url = f"s3://{self.bucket}/{s3_key}"
        
        # --- Step 2: 生成语义向量 (Embeddings) ---
        # vector = self.encoder.encode(image_path)
        # 这里模拟一个 512 维向量
        vector = [0.1] * 512 
        
        # --- Step 3: 存入 SQL (元数据) ---
        cursor = self.pg_conn.cursor()
        cursor.execute(
            "INSERT INTO assets (id, url, upload_time, user_id) VALUES (%s, %s, %s, %s)",
            (asset_id, s3_url, datetime.now(), metadata.get('user_id'))
        )
        self.pg_conn.commit()
        
        # --- Step 4: 存入向量数据库 ---
        # Payload 可以包含少量筛选用的元数据
        self.qdrant.upsert(
            collection_name=self.collection,
            points=[
                PointStruct(
                    id=asset_id,  # 使用相同的 ID 关联
                    vector=vector,
                    payload={"type": "image", "url": s3_url}
                )
            ]
        )
        print(f"Stored asset {asset_id} successfully.")

    def search(self, query_text):
        """
        多模态检索：文字 -> 向量 -> 图片
        """
        # 1. 把搜索词变成向量
        # query_vector = self.encoder.encode_text(query_text)
        query_vector = [0.1] * 512
        
        # 2. 在 Qdrant 中搜索
        hits = self.qdrant.search(
            collection_name=self.collection,
            query_vector=query_vector,
            limit=5
        )
        
        # 3. 返回结果 (包含 URL)
        return [hit.payload['url'] for hit in hits]

# 使用示例
# storage = MultimodalStorage()
# storage.save_image("cat.jpg", {"user_id": "u123"})
# results = storage.search("一只可爱的猫")
```

### 4. 前沿趋势：专用的多模态数据库

如果你不想自己维护这三套系统，可以使用**新一代多模态原生数据库**。

#### (1) LanceDB (强烈推荐用于本地/边缘侧)
*   **特点**: 这是一个无服务器（Serverless）、基于文件的向量数据库。
*   **优势**: 它支持**多模态数据列**。你可以直接把图片字节流存在 LanceDB 的列里，它自动管理存储和索引，不需要分离 S3 和 VectorDB。
*   **适用**: 个人开发者、本地 RAG 应用、自动驾驶数据管理。

#### (2) Deep Lake (Activeloop)
*   **特点**: 专为深度学习设计的数据湖。
*   **优势**: 它可以像 Numpy 数组一样流式传输多模态数据到 GPU。非常适合**训练数据**的存储和版本控制。
*   **格式**: 所有的图片、视频都被分块存储为 Tensor 格式，而非普通文件。

### 5. 总结与建议

*   **常规企业级应用**: 请坚持 **S3 + Postgres or mysql + Milvus/Elasticsearch** 的铁三角组合。这是最稳健、可扩展性最好的方案。
*   **单机/轻量级应用**: 尝试 **LanceDB**，开发体验极佳，一行代码解决存储和检索。
*   **视频检索特定场景**: 务必做好 **Chunking（切片）**，不要把整个视频变成一个向量，否则检索精度会非常差。


# 切片

切片（Chunking）是多模态应用（尤其是 RAG 和长上下文处理）中**最关键的数据治理环节**。

如果切片太粗，不仅会爆显存/上下文窗口，还会导致“大海捞针”时噪声太大（检索精度下降）；如果切片太细，语义支离破碎，模型无法理解上下文。

**切片的本质是：在物理大小限制（Token/Time/Pixel）和语义完整性（Meaning）之间寻找平衡。**

以下是针对**文本、图片、音频、视频**四大媒介的工业级切片策略：

---

## 1. 文本切片 (Text Chunking)
*虽然你问的是多模态，但文本通常是多模态的“锚点”，简单带过。*

*   **策略 A: 递归字符切分 (Recursive Character Split)** —— *最通用*
    *   **逻辑**: 先按段落 `\n\n` 切，太长就按句子 `.` 切，还长就按单词 ` ` 切。
    *   **工具**: LangChain `RecursiveCharacterTextSplitter`.
    *   **参数**: Chunk Size 500-1000 tokens, Overlap 10-20%.
*   **策略 B: 语义切分 (Semantic Chunking)** —— *进阶*
    *   **逻辑**: 利用 Embedding 模型计算相邻句子的相似度，相似度突变的地方就是“话题转换点”，在此处切刀。

---

## 2. 图片切片 (Image Chunking)
图片本身是 2D 空间数据。切片的目的是为了**突破分辨率限制**或**增强局部细节识别（如 OCR）**。

### 策略 A: 动态分辨率切片 (AnyRes / High-Res Cropping)
*这是目前 LLaVA-NeXT、Qwen-VL 等 SOTA 模型处理高清图的标准做法。*
*   **场景**: 输入一张 4K 的长图或报表，普通 resize 会导致文字模糊不可读。
*   **方法**:
    1.  **网格切割**: 将大图切成 $N$ 个 $336 \times 336$（模型原生分辨率）的子图。
    2.  **全局缩略图**: 额外生成一张整图的缩略图（Global Thumbnail）以保持全局视野。
    3.  **拼接**: 输入序列 = `[Global] + [Patch_1] + [Patch_2] ... + [Patch_N]`。
*   **代码思路**:
    ```python
    def crop_image(image, grid_size=(2, 2)):
        w, h = image.size
        # 切成 2x2 = 4 张子图
        sub_images = []
        for i in range(grid_size[0]):
            for j in range(grid_size[1]):
                box = (j * (w//2), i * (h//2), (j+1) * (w//2), (i+1) * (h//2))
                sub_images.append(image.crop(box))
        return sub_images
    # LLM 就能同时看清全局和局部细节
    ```

### 策略 B: 目标检测切片 (Object-centric Cropping)
*   **场景**: 监控摄像头、电商图。
*   **方法**: 先用 YOLO 或 DINO 检测出图中的关键物体（人、车、商品），把这些物体抠出来（Crop）单独作为切片存入向量库。

---

## 3. 音频切片 (Audio Chunking)
音频是 1D 时间序列。切片的难点在于**不要把一个单词或一句话切断**。

#### 策略 A: 基于静音切分 (VAD - Voice Activity Detection) —— *推荐*
*   **逻辑**: 检测波形中的静音片段。如果静音超过 500ms，就在这里切一刀。
*   **优点**: 保证了每一段都是完整的句子，不会切断单词。
*   **工具**: `silero-vad` 或 `pyannote.audio`。
*   **代码示例**:
    ```python
    # 伪代码：使用 silero-vad 获取语音时间戳
    timestamps = get_speech_timestamps(wav, model)
    # timestamps: [{'start': 0.5, 'end': 3.2}, {'start': 4.0, 'end': 8.1}]
    # 遍历时间戳切割音频文件
    ```

### 策略 B: 语义/对齐切分 (Transcript Alignment)
*   **逻辑**: 先用 Whisper 识别全文，得到带有时间戳的文字。根据文字的标点符号（句号）来反向切割音频。
*   **优点**: 语义绝对完整。
*   **缺点**: 需要先跑一遍 ASR，计算成本高。

### 策略 C: 固定滑动窗口 (Fixed Sliding Window)
*   **逻辑**: 每 30 秒切一段，重叠 5 秒。
*   **适用**: 背景音乐、环境音（没有明确的句子边界）。

---

## 4. 视频切片 (Video Chunking)
视频是“图片 + 音频 + 时间”的混合体，是切片最复杂的模态。

### 策略 A: 场景检测切分 (Scene Detection) —— *最符合人类直觉*
*   **逻辑**: 电影中镜头切换（Shot Change）时，通常意味着内容变了。计算相邻帧的直方图差异（Histogram Difference），超过阈值就切片。
*   **工具**: `PySceneDetect` (Python 库，基于 ffmpeg)。
*   **产物**: 一个 10 分钟的视频可能被切成 50 个“场景片段”。
*   **命令**:
    ```bash
    scenedetect -i video.mp4 detect-content list-scenes
    ```

### 策略 B: 固定时间间隔 (Fixed Interval)
*   **逻辑**: 简单粗暴，每 60 秒切一段。
*   **适用**: 会议录像、直播回放（画面变化不大，主要是人在讲）。

### 策略 C: 混合模态切分 (Hybrid Chunking)
*   **逻辑**:
    1.  **视觉流**: 每秒抽 1 帧。
    2.  **音频流**: 做 ASR 转录。
    3.  **对齐**: 以 ASR 的文字句子为基准。
        *   句子 A: 00:00 - 00:05 "现在我们要演示这个功能。" -> **切片 1 (包含这 5 秒的画面和声音)**
        *   句子 B: 00:05 - 00:12 "大家请看屏幕左上角..." -> **切片 2**
*   **优势**: 非常适合 RAG 检索（搜文字，定位视频片段）。

---

## 5. 总结：工业界最佳实践表

| 媒介          | 最佳切片策略                  | 核心工具/算法           | 适用场景             |
| :---------- | :---------------------- | :---------------- | :--------------- |
| **文本**      | **递归字符切分 (Recursive)**  | LangChain         | 绝大多数文档 RAG       |
| **图片**      | **AnyRes 网格切分**         | Pillow / OpenCV   | 包含密集文字的报表、高清地图   |
| **音频**      | **VAD 静音检测**            | Silero VAD        | 播客、会议录音、客服电话     |
| **视频 (通用)** | **场景检测 (Scene Detect)** | PySceneDetect     | 电影、电视剧、广告片       |
| **视频 (会议)** | **ASR 语义对齐**            | Whisper Timestamp | Zoom/腾讯会议录像，教学视频 |

## 6. 存储时的元数据设计 (Metadata Design)

切片后，**千万不要丢失“它是从哪来的”这个信息**。在存入向量数据库时，必须带上元数据：

```json
// 一个视频切片的存储结构示例
{
  "id": "chunk_v892_05",
  "vector": [0.12, -0.5, ...], // 多模态向量
  "payload": {
    "source_url": "s3://bucket/video_tutorial.mp4",
    "modality": "video",
    "start_time": 125.5,   // 关键：切片开始时间
    "end_time": 138.0,     // 关键：切片结束时间
    "transcript": "现在点击红色的按钮...", // 对应的文字
    "prev_chunk_id": "chunk_v892_04", // 链表结构，方便在大模型里回溯上下文
    "next_chunk_id": "chunk_v892_06"
  }
}
```

这样，当 LLM 检索到这个切片时，它不仅知道“这里有相关内容”，还能知道“这是视频的第 2 分 05 秒”，并在播放器里直接跳转到这一秒。


# 怎么输出

多模态输出（Multimodal Output/Generation）在工程实现上通常分为两条截然不同的技术路线：**“原生（Native）路线”** 和 **“代理（Agent/Pipeline）路线”**。

对于绝大多数开发者而言，**代理路线（LLM + Tools）是目前最成熟、成本最低且可控性最好的方案。**

以下是详细的架构拆解和实现指南。

---

### 1. 两大核心技术路线

#### 路线 A：代理/工具调用 (The Agentic Approach) —— *工业界主流*
*   **原理**: LLM 是“大脑”，它不直接画图或发声，而是生成**指令（Function Call）**，指挥“手脚”（专用的生成模型）去执行。
*   **流程**: `用户指令 -> LLM 判断意图 -> LLM 生成绘图Prompt -> 调用 Stable Diffusion -> 返回图片 URL -> 展示给用户`。
*   **优势**: 模块化，想换更好的绘图模型随时换；可控性高。
*   **劣势**: 有割裂感，推理并非端到端。

#### 路线 B：原生生成 (The Native Approach) —— *前沿/SOTA*
*   **原理**: 模型输出的 Token 序列中，既包含文字 Token，也包含图像/音频 Token。
*   **代表**: GPT-4o (Audio-out), Chameleon (Image-out)。
*   **优势**: 极低延迟，情感丰富，图文/音文融合极佳。
*   **劣势**: 训练极难，闭源为主，开源模型（如 Emu）部署成本极高。

---

### 2. 各种模态的具体输出方案 (Agentic 模式)

我们主要关注**代理模式**，因为这是你现在就能写代码实现的。

#### (1) 图片输出 (Image Generation)
*   **核心逻辑**: Text-to-Image (T2I)。
*   **推荐模型/API**:
    *   **开源/自部署**: **Flux.1** (目前最强开源), **Stable Diffusion XL (SDXL)**, **Kolors (可图)**。
    *   **API**: Midjourney (非官方), DALL-E 3, Black Forest Labs (Flux API)。
*   **实现方式**:
    1.  LLM 识别用户想画图。
    2.  LLM 优化 Prompt（例如把“画只猫”改写为“一只毛发蓬松的布偶猫，4k画质，电影布光”）。
    3.  调用 API 生成图片，保存到 S3。
    4.  返回 Markdown 格式：`![image](https://s3.bucket/cat.jpg)`。

#### (2) 音频输出 (Audio Generation / TTS)
*   **核心逻辑**: Text-to-Speech (TTS) 或 Text-to-Music。
*   **推荐模型/API**:
    *   **语音 (TTS)**:
        *   **ChatTTS**: 适合对话，有笑声、停顿，韵律自然。
        *   **CosyVoice (阿里)**: 支持极速复刻声音，情感控制极佳。
        *   **F5-TTS**: 非自回归架构，速度极快。
    *   **音乐**: **Suno**, **Udio** (通常通过爬虫或逆向 API 调用)。
*   **关键点**: **流式输出 (Streaming)**。音频生成慢，必须一边生成字节流，一边推给前端播放，否则用户等待感太强。

#### (3) 视频输出 (Video Generation)
*   **核心逻辑**: Text-to-Video。目前是**异步任务**，不能实时。
*   **推荐模型**:
    *   **开源**: **CogVideoX (智谱)**, **HunyuanVideo (腾讯)** —— *目前开源界最强*。
    *   **商业 API**: Runway Gen-3, Kling (可灵), Luma Dream Machine。
*   **实现方式**:
    1.  用户提交请求。
    2.  后端加入任务队列 (Celery/Redis Queue)。
    3.  前端轮询 (Polling) 状态。
    4.  任务完成后，返回 MP4 链接。

---

### 3. 代码实战：构建多模态输出 Agent

这是一个基于 **Function Calling** 思想的 Python 伪代码，展示如何让一个纯文本 LLM 输出图片和语音。

```python
import json
from openai import OpenAI  # 假设使用兼容 OpenAI 接口的模型
# 假设引入了外部工具库
# from tools import generate_image_flux, generate_speech_cosyvoice

class MultimodalOutputAgent:
    def __init__(self):
        self.client = OpenAI(api_key="...", base_url="...")
        
        # 定义工具：告诉 LLM 它有哪些能力
        self.tools = [
            {
                "type": "function",
                "function": {
                    "name": "generate_image",
                    "description": "当用户想要看图、画图时调用此函数",
                    "parameters": {
                        "type": "object",
                        "properties": {
                            "prompt": {"type": "string", "description": "用于绘图的详细英文描述词"},
                        },
                        "required": ["prompt"]
                    }
                }
            },
            {
                "type": "function",
                "function": {
                    "name": "speak_text",
                    "description": "当用户明确要求朗读或用语音回复时调用",
                    "parameters": {
                        "type": "object",
                        "properties": {
                            "text": {"type": "string", "description": "要朗读的文本内容"},
                            "emotion": {"type": "string", "enum": ["happy", "sad", "angry"]}
                        }
                    }
                }
            }
        ]

    def chat(self, user_query):
        # 1. LLM 思考与决策
        response = self.client.chat.completions.create(
            model="gpt-4-turbo",
            messages=[{"role": "user", "content": user_query}],
            tools=self.tools,
            tool_choice="auto" 
        )
        
        msg = response.choices[0].message
        
        # 2. 检查是否触发了工具 (多模态输出)
        if msg.tool_calls:
            for tool_call in msg.tool_calls:
                func_name = tool_call.function.name
                args = json.loads(tool_call.function.arguments)
                
                # --- 分发到具体的模态生成器 ---
                
                if func_name == "generate_image":
                    print(f"正在调用 Flux 画图: {args['prompt']}...")
                    # image_url = generate_image_flux(args['prompt'])
                    image_url = "https://s3.example.com/generated_cat.jpg" # 模拟返回
                    return f"好的，这是为您生成的图片：\n![image]({image_url})"
                
                elif func_name == "speak_text":
                    print(f"正在合成语音 ({args.get('emotion')}): {args['text']}...")
                    # audio_url = generate_speech_cosyvoice(args['text'])
                    audio_url = "https://s3.example.com/speech.mp3"
                    return f"语音已生成：<audio controls src='{audio_url}'></audio>"
        
        # 3. 如果没有工具调用，直接返回文本
        return msg.content

# 使用示例
agent = MultimodalOutputAgent()

# 测试图片输出
print(agent.chat("给我画一只赛博朋克风格的猫"))
# 输出: 好的，这是为您生成的图片：\n![image](https://s3...)

# 测试语音输出
print(agent.chat("请用开心的语气读这句话：你好世界"))
# 输出: 语音已生成：<audio controls src='...'></audio>
```

---

### 4. 前端展示 (Rendering)

后端生成了 URL，前端（Web/App）需要正确渲染。

*   **Markdown 渲染 (最通用)**:
    大多数 Chat UI（如 Streamlit, ChatGPT, LobeHub）都支持 Markdown。
    *   图片: `![Alt](url)` -> 自动渲染为 `<img>`
    *   音频: 需要 HTML 标签 `<audio src="url" controls />` (部分 Markdown 引擎支持 HTML)。
    *   视频: `<video src="url" controls />`。

*   **特殊处理：流式音频 (Streaming Audio)**:
    为了实现类似 GPT-4o 的“边说边听”体验，不能等生成完整个 MP3 再发给前端。
    *   **后端**: 使用 WebSocket 或 SSE (Server-Sent Events) 发送音频的 **Chunk (字节块)**。
    *   **前端**: 使用 `AudioContext` 或 `MediaSource Extensions (MSE)` 接收二进制流并实时播放。

### 5. 总结：多模态输出工程清单

| 需求 | 推荐方案 (2025年标准) | 关键技术点 | 延迟预期 |
| :--- | :--- | :--- | :--- |
| **生图** | **Flux.1 (Dev/Schnell)** | Prompt 增强, LoRA 风格微调 | 2s - 10s |
| **语音 (TTS)** | **CosyVoice / ChatTTS** | 流式传输 (Streaming), 情感控制 | < 500ms (首包) |
| **视频** | **HunyuanVideo / CogVideoX** | 异步任务队列, GPU 显存优化 | 2min - 10min |
| **音乐** | **Suno API** (非官方) | 歌词对齐 | 30s+ |

**核心建议**: 不要试图把所有模型都塞进一个显卡里。**多模态输出本质上是一个微服务架构**。LLM 是调度员，Image-Worker 和 Audio-Worker 应该部署在不同的 GPU 机器上，通过 API 协同工作。

def generate_rag_response(user_query, retrieved_texts, retrieved_images):
